Inventory Customer Management GUI

Khoi Ngo

In this project, I redesigned an inventory management system into a distributed system using the client-server architectural pattern and using a Model-View-Controller design pattern. The objectives of this project are:

  1. Conceptual database design based on a narrative
  2. Logical database design through relational model mapping
  3. Logical database design through relational model mapping
  4. Java Graphical User Interface Event-Handling
  5. Client-Server Architecture
  6. Working with database through a Java program
  7. MVC

I used the narrative given to start building my database. At the start of the design, I looked for all the entities participating. This included Tool, Supplier, Customer, and Order. After, I added the specialization to some of the entities and the specializations that can not be overlapping, either the entity is one or the other. For Tool, I assumed that a tool must be either Electrical or Non-Electrical and similar to supplier; it had to be either International or Local. However for Customers there is partial participation because from the requirement description, it said the Customers “can” be either residential or commercial. So I assumed that the Customer could be either or none. Next, I added the relationships. Tools must be from a Supplier and Supplier must make a Tool. A Customer must buy a tool but a Tool does not have to be bought. This relationship will have a relation later so to differentiate duplicate tuples, I will add a date attribute. Tool, Supplier, and Order must participate since the order line’s primary key depends on Supplier, Tool, and Order information. Figure 1 displays the UML database design  I created.

After finishing the conceptual design, the next step was to create the logical design of how the database will be built. For the specialization, I decided to use 8C where we add the specializations in the entity relations. Since the specializations only have 1 or 0 attributes, I found it would be a waste to separate the specialization relations. Therefore, I combine the specializations in their respective entities. I assume that the specializations are equal to limit the number of NULL values that may be present. After following the steps to transfer between conceptual to logical designs, Figure 2 shows how the database will be built with 6 tables.

After designing and building the database, I started designing the architecture of the front and back ends. I used a Model-View-Controller design pattern for both the client and server side. The controllers are the only point of contact for each model and view. The user's view notifies the client controller which then changes the client model. Then the client controller sends the information to the server controller to change its version of the model. It will also update the database. The client and server communicate via socket and sends the objects to change. The UML class diagram is designed in Figure 3. 

The front-end is a Java GUI. The main screen will have to options for the user, the customer and tools management system. The user will select the options to move to a different view. Figure 4 shows the main screen.

The customer management system handled the customers that have ordered from the store. The GUI has two basic features, displaying the customer's information and searching for customers. To search, the user will need to specify what category to search by. The options are by customer id, customer name, or customer type. Once an option is selected, the user inputs a keyword and press search. A list of search results will be displayed below. The user can select a customer and their information will populate the customer information fields. The user can edit the information. The user can also add a new user or delete a user. Figure 5 shows the GUI of the customer management system. 

The tool management system allows the user to learn and buy tools. The system is similar to the previous one. The user can search for tools by id or name. The user can select a tool from the search results to see more information about it. The information will fill the tool information fields. Unlike the previous system, the user cannot edit the tool, but they can still add a new tool or delete the tool. The user can purchase the tool and it will decrease the stock of the item by one. Another feature is the user can check the tool inventory. When the check inventory button is pressed, the system will check the stocks of each tool. If a tool is below the stock threshold, the system will place an order for the item to be restocked. The user can view orders by inputting the date of the desired order and press the print order button. The order will be saved as a text file. The GUI for the tool management system is shown in Figure 6.

Summary of Features

  • Customer Management System
    • Search for a user by id, name or customer type
    • Display user's information
    • Edit and save user information
    • Add a new user
    • Delete a new user
  • Inventory Management System
    • Check tool inventory, restock any item under stock threshold
    • Print order of certain date, saves as a text file of which items need to be restocked
    • Search for a tool by id or name
    • Display tool's information
    • Add a new tool
    • Delete a tool
    • Buy a tool, decreases stock of item
